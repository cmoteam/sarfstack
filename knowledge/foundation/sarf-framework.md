# SARF — AI時代のマーケティングフレームワーク

## 定義

SARFは、AIエージェントと協働してマーケティングを推進するための基本サイクルを4段階で表したフレームワーク。
SARFStackの全スキル・全ワークフローの基底にあるOSであり、個別のフレームワーク（AARRR・STP・RAM-CE等）の上位に位置する「AIとの関わり方の型」を定義する。

読みは「**サーフ**」。波乗りのメタファーと整合する:
- **Set**（セット）= 波待ちのポジション取り
- **Ask**（アスク）= どの波に乗るか見極める
- **Release**（リリース）= 波に放たれる＝本番環境へ反映する
- **Feedback**（フィードバック）= 乗った結果を読み、次のセットに備える

```
Set ─→ Ask ─→ Release ─→ Feedback ─┐
 ↑                                    │
 └────────────────────────────────────┘
```

| 段階 | 意味 | 人間の役割 | AIの役割 |
|------|------|-----------|----------|
| **Set** | 自分の情報をAIに伝える | 事業・ICP・ブランド・数字・制約を言語化して渡す | 文脈を吸収し、前提を明示的にする |
| **Ask** | AIとのコミュニケーション | 何を、誰の視点で、どこまで出させるかを問いとして組む | 複数の選択肢・根拠・抜け漏れを返す |
| **Release** | 判断して本番反映 | 出力の中から取捨選択し、責任を持って本番環境にリリースする | 実装物・コピー・コードをそのまま使える粒度で出力する |
| **Feedback** | 結果をAIに戻す | 実行後の数字・反応・違和感を記述して戻す | 次のSetを更新し、リファインして精度を上げる |

## なぜSARFが必要か

従来のマーケティングフレームワーク（AARRR・STP・4P等）は**「何を考えるか」**の地図を提供してきた。
一方、AIエージェントと協働する時代においては**「どうやって考えさせるか」**そのものが新たな競争力になる。

- **Setが弱い組織** → AIが汎用回答しか返せない。どこの会社でも通用する薄いアウトプットになる
- **Askが弱い組織** → 良い情報を持っていても引き出せない。「いい感じにして」で止まる
- **Releaseが弱い組織** → AIの提案を眺めるだけで終わる。PDCAの「D」が抜ける
- **Feedbackが弱い組織** → 同じミスを繰り返す。AIが賢くならず、ナレッジが蓄積されない

SARFStackの知識ベース構造（`knowledge/foundation` `knowledge/latest` `memory/company` `memory/results`）は、SARFの**Set**を継続的に強化するための仕組みである。

## 4つの段階の詳細

### S — Set（前提を渡す）

**原則**: AIは渡された情報の範囲でしか判断できない。出力の質は Set の質で9割決まる。

渡すべき情報:
- **事業**: 何を売っているか、収益モデル、現状の数字（`memory/company/company-overview.md`）
- **ICP**: 誰に売っているか、その人の1日・課題・情報収集経路（`memory/company/icp.md`）
- **ポジショニング**: 競合との差別化軸（`memory/company/positioning.md`）
- **ブランド**: トーン&マナー、使ってよい言葉・避ける言葉（`memory/company/brand-guidelines.md`）
- **制約**: 予算・期間・人的リソース・法規制
- **直近データ**: 最近のパフォーマンス、成功/失敗の履歴（`memory/results/performance-data.md`）

Setが成立しているかのチェック:
- [ ] `[TODO]` が残っていないか
- [ ] 「弊社は」ではなく具体的な固有名詞・数値で書かれているか
- [ ] 成功事例だけでなく失敗事例・避けたい表現も書かれているか

### A — Ask(問いを組む)

**原則**: 漠然とした問いには漠然とした答えしか返らない。問いの解像度が成果物の解像度を規定する。

良いAskの構造:
1. **誰の視点で**（CEO / CMO / SEO / ICP本人 / 新規訪問者）
2. **何を**（評価 / 制作 / 分析 / 比較 / 予測）
3. **どの深さで**（案出し3つ / 完成品 / レビュー付き / A/Bパターン）
4. **何を判断基準に**（ICE / ROI / ブランド整合 / CVR予測）
5. **どの形式で**（表 / チェックリスト / コード / 原稿）

SARFStackでは、スラッシュコマンドがこのAskをテンプレート化している:
- `/ask-ceo` = 「CEO視点で、ROI基準で、GO/PIVOT/KILLの判定付きで評価せよ」
- `/ui-designer` = 「UI/UX視点で、CVR改善基準で、修正後コード込みでレビューせよ」

#### Ask のサブタイプ: Design Ask / Review Ask

同じ「Ask」でも、何もない状態から設計させる問いと、既存成果物を評価させる問いは性質が異なる。サイクルを回す際は両者を明示的に区別する。

- **Design Ask** — ゼロから設計させる問い（戦略立案、ICP仮説、LP構成案、キーワード戦略）。Set 情報をもとに **新しい選択肢を生成** させる。Release の前段で使う。
- **Review Ask** — 既存の成果物を評価させる問い（コピー品質、LPのCVR観点、広告の戦略整合性、経営インパクト）。Set 情報を判定基準として **既存物をジャッジ** させる。Release 直前のゲートとして使う。

ワークフローの中では `Design Ask → Release（叩き台生成） → Review Ask → Release（本番反映）` のように Ask-Release が入れ子で回るのが通常。各スキルの SARF Alignment に `Ask Subtype` が記載されている。

| スキル | Design | Review |
|--------|--------|--------|
| `/ask-cmo` | 戦略・チャネル設計 | 成果物の戦略整合性 |
| `/ask-ceo` | — | GO / PIVOT / KILL |
| `/creative-director` | — | APPROVE / REVISE / REWORK |
| `/seo-specialist` | キーワード戦略 | 技術監査・既存コンテンツ評価 |
| `/ask-consultant` | ゼロベース再設計提案 | 前提・聖域・戦略のゼロベース率直レビュー |
| `/ui-designer` | — | CVR観点のUI評価 |
| `/estimate` | スコープ設計・工数積算 | — |

**選び方の指針**: 「何もない状態からアイデアを出させたい」なら Design Ask、「手元の成果物を通すか判定したい」なら Review Ask。混ぜると評価軸がブレる。

### R — Release（判断して本番反映）

**原則**: AIの出力は「叩き台」。最終的な意思決定と本番反映の責任は人間が持つ。

Releaseの段階で人間がやること:
- **選ぶ**: 複数案の中から、自社の文脈に最も合うものを選択する
- **削る**: AIが出しがちな冗長な要素・汎用的な装飾を削る
- **足す**: AIが持ちえない現場の暗黙知・1次情報を足す
- **リリースする**: 原稿を公開する、広告を入稿する、LPをデプロイする

「AIに書かせて終わり」「レビューレポートを眺めて満足」は Release ではない。**本番環境に反映されて初めてRelease完了**。

### F — Feedback（結果を戻す）

**原則**: AIを賢くするのはあなたの実データ。Feedbackのないプロジェクトは、毎回ゼロから始まる。

戻すべきフィードバック:
- **数字**: CTR / CVR / CPA / LTV の実績 vs 予測
- **定性**: 顧客からの反応、営業現場で聞かれた質問、クレーム
- **差分**: AIの提案のうち、採用したもの / 却下したもの / 修正したもの
- **学び**: なぜ予測と実績がズレたか。次回のSetに何を追加すべきか

Feedbackの反映先:
- `memory/results/performance-data.md` — 企業固有の実績数値（gitignore 対象）
- `memory/company/*.md` — 検証された知見（ICPの解像度上昇、ポジショニングの微修正／gitignore 対象）
- `knowledge/latest/industry-trends.md` — 外部観測として一般化できるもの（共有可）

## SARFとSARFStackの対応

```
┌───────────────────┬──────────────────┬──────────────────┬──────────────┐
│  Set              │  Ask             │  Release         │  Feedback    │
├───────────────────┼──────────────────┼──────────────────┼──────────────┤
│ memory/company    │ /ask-cmo         │ /contents-editor │ /data-analyst│
│ memory/results    │ /ask-ceo         │ /ads-manager     │ /weekly-     │
│ knowledge/latest  │ /seo-specialist  │ /flow-landing-   │  retro       │
│ knowledge/        │ /creative-       │  page            │              │
│  foundation       │  director        │ （成果物出力＋   │ → Setに還元  │
│                   │                  │   本番反映）     │              │
└───────────────────┴──────────────────┴──────────────────┴──────────────┘
```

- **Set層** = `knowledge/`（base） + `memory/`（per-project）（常に最新化する）
- **Ask層** = レビュー系スキル（問いの型を提供する）
- **Release層** = 制作系スキル / ワークフロー（実装物を出力し、本番に反映する）
- **Feedback層** = 分析系スキル（結果を測り、知識ベースに還元する）

## アンチパターン

- **Set丸投げ**: 「とにかくいい感じで」— AIは平均的な回答しか返せない
- **Askの過剰**: 1つのプロンプトで10個聞く — どれも中途半端になる。1サイクル1目的
- **Release不在**: レビューレポートを読んで満足 — 本番反映がなければ何も変わらない
- **Feedbackの断絶**: 出した施策の結果を戻さない — 次回も同じ精度の出力しか得られない
- **Set汚染**: 検証されていない仮説を事実としてmemory/companyに書く — 以降全ての出力が歪む

## Practitioner's Checklist

新しい施策に着手する前に以下を確認:

- [ ] **Set**: 該当スキルの `Required Knowledge` ファイルは最新か？ `[TODO]` が残っていないか？
- [ ] **Ask**: どの視点・何を判断基準に・どの形式で出させるか、スラッシュコマンドを明示しているか？
- [ ] **Release**: 誰が・いつまでに・どこに本番反映するかが決まっているか？
- [ ] **Feedback**: 何の数字を・いつ測って・どこに書き戻すかが決まっているか？

このチェックが通らないうちは、AIに投げても成果は出ない。SARFは省略できない。
