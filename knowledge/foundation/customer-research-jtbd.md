# Customer Research & JTBD — 顧客リサーチとJobs to be Done

マーケティング施策の根拠は**常に顧客の言葉**にある。フレームワーク・データも重要だが、N1の生々しい発言を聞いていないマーケは漂流する。
顧客リサーチはSAAFの **Set** を更新する最重要活動であり、N1インタビューの内容は必ず `memory/company/icp.md` に反映させる。

## なぜJTBD（Jobs to be Done）か

従来のペルソナ分析の限界：
- 「30代女性・都内在住・年収600万」= **同じ人でもジョブが違えば行動が違う**
- デモグラでは行動を予測できない

JTBDの核心：
> 「顧客はプロダクトを買うのではない。**人生で達成したい進歩（Progress）**のためにプロダクトを"雇う（Hire）"」
> — Clayton Christensen

ミルクシェイクの例：
- 同じミルクシェイクでも**朝は「通勤中の退屈しのぎ」**、**午後は「子供へのご褒美」**という別ジョブで雇われている
- デモグラで分析すると見逃す。**状況 × ジョブ**で捉える

## JTBDのフレーム

### 基本フォーマット
```
When I [状況],
I want to [動機・達成したいこと],
So I can [期待する成果・より大きな目的].
```

### 具体例
```
When I: 朝9時からのMTGに向かう通勤中、
I want to: 片手で食べられて時間が潰せるものが欲しい、
So I can: 退屈せず、気持ちを整えて出社できる。
```

### Forces of Progress（4つの力）

```
現状維持の力                    新しい解決策への引力
─────────────────                ─────────────────
  ② 既存への惰性          →      ① 現状への不満（Push）
  ④ 新しいものへの不安    →      ③ 新しい解決策の魅力（Pull）
```

**Push > Pull > 惰性 + 不安** のときに購入が起きる。

マーケティングの仕事は：
- Push を言語化する（「今のままで本当にいいですか？」）
- Pull を魅力的に見せる（ビフォアフ、事例）
- 惰性を壊す（締切、限定性）
- 不安を取り除く（保証、返金、オンボーディングサポート）

## Switch Interview（スイッチインタビュー）

JTBDの中核メソッド。**「購入を決めた直近の顧客」**に、
**採用前後の時系列を詳細に再現**してもらうインタビュー。

### 6つの聞くべきタイミング

```
First Thought ──→ Passive Looking ──→ Active Looking
  (最初に思った瞬間)  (なんとなく調べ始め) (本気で探し始め)
        ↓
Event 1 ──→ Event 2 ──→ Decision ──→ First Use ──→ Satisfaction
 (きっかけ1)  (きっかけ2)   (購入決定)     (初使用)      (満足/後悔)
```

特に重要な4つの問い：
1. **「最初にこれが必要かもと思ったのは、いつ、何があったからですか？」** → First Thought
2. **「本気で探し始めた具体的なきっかけは何ですか？」** → Active Looking Trigger
3. **「何と比較しましたか？他の候補はなぜダメでしたか？」** → Competitive Set（競合は自社が思う競合とは限らない）
4. **「買う直前に、最後に払拭された不安は何でしたか？」** → Anxieties

### Switch Interviewが強い理由
- **記憶は状況に紐づいている** — 「いつ」「どこで」「誰と」を聞くと芋づる式に思い出される
- **本人の意志より"出来事"を聞く** — 「なぜ買ったか」は後付け合理化、「何があったか」は事実
- **競合が明らかになる** — 顧客から見たSwitch元（Hire from）がわかる
- **Desired Progressが浮かぶ** — 機能ではなく「何を達成したかったか」が見える

### 進め方
- N=**10〜15件**で十分（Saturation: 新しい発見が出なくなる点）
- 1件 **45〜75分**、録音必須
- タイムラインを可視化しながら聞く（ホワイトボード / Miro）
- **オープンクエスチョン**中心、誘導質問を避ける

## The Mom Test — 聞き方の罠を避ける

Rob Fitzpatrick *The Mom Test* の3原則：

1. **未来の話ではなく、過去の話を聞く**
   - ❌「もしこの機能があったら使いますか？」→ 全員Yesと言う
   - ✅「最近、似たような問題をどう解決しましたか？」

2. **意見ではなく、事実と行動を聞く**
   - ❌「この価格、妥当だと思いますか？」→ 気を使って「妥当」と言う
   - ✅「今このカテゴリに月いくら払っていますか？」

3. **自分のアイデアの話をしない**
   - アイデアを話すとフィードバックが歪む。**顧客の生活の話だけを聞く**

## N1分析（西口一希）

- 1人の顧客を**徹底的に理解する**ほうが、100人のアンケートより深い洞察が得られる
- 「**便益 × 独自性**」のマトリクスでアイデアを評価
  - 便益: あると嬉しい / 独自性: 他にはない
  - 両方高い = 独自便益（強いアイデア）
  - 便益のみ = コモディティ
  - 独自性のみ = 独りよがり
  - 両方なし = 無価値

### 5つの顧客セグメント（9セグマップ）
認知 × ブランド選好 × 購買頻度 で顧客を9分類し、**どのセグメントからどのセグメントへ動かすか**を施策の単位にする。

1. **ロイヤル顧客**（認知あり・選好あり・高頻度購買）
2. **一般顧客**（認知あり・選好あり・低頻度購買）
3. **離反顧客**（認知あり・選好あり・購買なし）
4. **認知・選好・未購買**
5. **認知・非選好・未購買**
6. **未認知**

## ProblemエリアとSolutionエリアの区別

PMF前の最大の失敗：**Solutionに恋して、Problemを確認しない**。

- **Problemフェーズ** — 「そもそもこの課題は存在するか / 深いか / お金を払う痛みか」
  - アウトプット: Problem文言、定量的な痛みの大きさ（時間・金額・頻度）
- **Solutionフェーズ** — 「我々の解決策が、他の手段より良いか」
  - アウトプット: プロトタイプへの反応、WTP（支払意思額）

Problemが浅いのにSolutionを磨いても、**誰も買わないプロダクト**ができる。

## 定性リサーチの種類と使い分け

| 手法 | 向いている目的 | N数 | 時間 |
|------|--------------|-----|------|
| **Switch Interview** | 購入意思決定の解明 | 10〜15 | 60min |
| **Diary Study** | 日常での使用文脈理解 | 5〜10 | 数日〜週 |
| **Usability Test** | プロダクト改善 | 5〜8 | 30min |
| **フォーカスグループ** | 集団ダイナミクス / ブランド認識 | 6〜8人 × 複数組 | 90min |
| **Observational（行動観察）** | 言語化されない行動の把握 | 5〜10 | 可変 |
| **Contextual Inquiry** | 現場での実際の使われ方 | 5〜10 | 60〜120min |

## 定量リサーチとの組み合わせ

定性で**仮説発見**、定量で**規模検証**。

```
定性（Switch Interview） → 仮説（〇〇というジョブがある）
    ↓
定量（サーベイ n=300+） → 検証（市場の何%がそのジョブを持つか）
    ↓
施策立案
```

片方だけでは判断できない：
- 定性のみ = 「面白い話だけど、市場があるか不明」
- 定量のみ = 「数字は動いたけど、なぜかわからない」

## サーベイ設計の落とし穴

- **5段階リッカート**は中央（3）に集中しやすい → 偶数段階（4 or 6）が判別力高い
- **NPS単独**では改善アクションに落ちない → 必ず自由記述+ドライバー設問を併置
- **順序効果** — 設問の順序で回答が変わる → ランダマイズ
- **Leading Question** — 「この優れた機能について…」= 誘導

## リサーチの成果物

インタビューや分析は**必ず以下の形で資産化**する：

1. **Customer Journey Map** — 購入前〜利用中〜離脱までの時系列
2. **JTBDステートメント集** — 発見されたジョブのリスト
3. **Voice of Customer（VOC）** — 印象的な生発言の集積（コピーの源泉）
4. **競合比較表** — 顧客視点のSwitch元 / 評価軸
5. **ICPアップデート** — `memory/company/icp.md` を書き換える

**聞いて終わり、書いて終わりにしない。** コピー・広告・LP・プロダクトに反映して初めて完了。

## SAAFとの接続

- **Set**: インタビューから得たJTBD / 競合 / 反論 / Anxietiesを `memory/company/icp.md` `competitors.md` に書く
- **Ask**: 「このSwitch Interviewログから主要ジョブを3つ抽出して」「Problem検証のインタビュースクリプトを作って」
- **Action**: LP・広告・メールのコピーをVOCの生発言から作り、テストする
- **Feedback**: 施策の結果（CVR・反応）を、どのジョブ仮説が当たっていたかの検証として記録
